🎯The Problem with Naive Retrieval

Why BM25 and vector search are not enough for structured technical documents.

📦 BM25 Keyword Retrieval

Returns sentences with the most keyword overlap. Completely blind to semantic relationships. A sentence mentioning "PostgreSQL" and one explaining why PostgreSQL was chosen are scored identically.

🔢 Vector Similarity

Better at semantic meaning but treats each sentence independently. Retrieves the most similar sentences but breaks causal chains — the "why" and the "what" end up disconnected.

🧠 SCRE

Understands types of sentences (decision, constraint, reason) and follows the graph of relationships between them. Retrieves not just relevant sentences but their complete reasoning context.

⚠️
The core insight:

When a user asks "why did we choose X?", they need the decision and the reasons and the constraints that drove it. BM25/Vector find the decision but drop the reasoning chain. SCRE follows the graph.

⚙️Six-Stage Pipeline

Every call to ingest() + retrieve() runs these stages.

1

Ingestion & Intelligent Parsing

Raw text is split into paragraphs and sentences. Multi-line Label:\nValue patterns are merged to prevent fragmentation. Markdown lists, code blocks, and header hierarchies are preserved. Ordered list sequences are detected as potential workflow candidates.

2

Semantic Unit Classification

Each sentence is passed through a configurable strategy chain: (1) SDLC regex matcher for tagged items like [DEC-01], (2) Markdown key-value extractor, (3) spaCy NLP dependency parser as fallback. The result is a typed SemanticUnit with subject, relation, object, and category.

3

Dual Graph Construction

Two graphs are built simultaneously: a Knowledge Graph (entity → sentence index adjacency list) and a Reasoning Graph (directed causal edges between sentence indices). The reasoning graph detects lexical markers like "because", "requires", "satisfies", "depends on".

4

Query-Aware Hybrid Scoring

Every semantic unit is scored against the incoming query using a composite formula combining dense embedding similarity (cosine via all-MiniLM-L6-v2), IDF-weighted lexical overlap, named entity matching, noun chunk matching, graph connectivity, intent alignment, and hard identifier boosting.

5

Context Expansion

Top-K selected units are expanded in two passes: (1) Adjacent context window — ±N neighbours to resolve pronouns and preserve local coherence; (2) Reasoning chain traversal — 2-hop graph walk to pull in causes, consequences, and constraints linked to selected decisions.

6

Final Assembly & Metrics

De-duplicate by Jaccard similarity (threshold 0.85). Re-order by original sentence index to preserve narrative flow. Inject section context headers to prevent detachment. Return clean context string + full compression metadata.

🔬Semantic Unit Extraction

How SCRE turns raw sentences into typed, structured semantic units.

Recognized Semantic Types

decision constraint reason workflow fact goal task outcome comparison

Category Weights

Each type has a baseline priority weight applied before query-specific scoring:

constraint decision reason

Weight 1.0 — Highest priority. These carry the core semantics of structured documents.

goal workflow

Weight 0.95 — High priority. Purpose statements and procedural flows are critical for comprehension.

task outcome fact

Weight 0.70–0.90 — Supporting information. Facts (fallback) have lowest base weight.

Workflow Detection

💡
Special case: ordered procedure detection.

When a non-list sentence is immediately followed by one or more numbered/bulleted list items, SCRE collapses them into a single WorkflowUnit. This preserves procedure integrity — the header + all steps are treated as one atomic semantic unit and represented as a structured JSON object.

🕸️Dual Graph Construction

Two complementary graph structures power SCRE's retrieval intelligence.

📌 Knowledge Graph

An entity → sentence-index adjacency list. Built from subject and object fields of all SemanticUnits. A unit with many entity co-occurrences (high graph connectivity) gets a relationship distance bonus during scoring — it's a hub of important information.


graph["postgresql"] = {3, 7, 12, 18}

"PostgreSQL" appears in sentences 3, 7, 12, 18 → these are densely connected to any query about PostgreSQL.

🔗 Reasoning Graph

A directed edge set between sentence indices representing causal relationships. Links are created when:

  • A reason unit follows a decision
  • A task follows a goal
  • Lexical causal markers detected: because, enables, depends on, satisfies, violates, fulfills…

graph[7] = {3, 12} → sentence 7 is causally linked to 3 and 12.

Top-K Selection with Token Floor

After scoring, units are ranked descending. Selection continues until max_sentences is reached, but only stops early if min_tokens is also satisfied. This prevents under-retrieval for short documents.

⚡
De-duplication during selection:

Before adding a unit to the selected set, its Jaccard similarity against all already-selected units is computed. If similarity > 0.85, the unit is skipped. This prevents semantically redundant sentences from consuming the token budget.

🔭Context Expansion

After top-K selection, SCRE expands the result to ensure coherence and complete reasoning chains.

Adjacent Context Window

For every selected sentence at index i, all sentences at indices i-window to i+window are added to the result set (if not already present). Default context_window=1.


Why: Pronouns like "it", "this", "they" in a selected sentence only make sense if the preceding sentence is included. The window ensures local coreference resolution.

Reasoning Chain Expansion (2-Hop)

For every selected sentence index, SCRE performs a 2-hop BFS on the reasoning graph:

selected = {7}
hop1 = graph[7] = {3, 12}
hop2 = graph[3] ∪ graph[12] = {1, 18}
final = {1, 3, 7, 12, 18}

🧪Benchmark Setup

Research-grade evaluation across 75 real-world structured technical documents.

Document Corpus

  • 25 Kubernetes KEPs — Enhancement Proposals with Motivation, Design, Alternatives, Goals, Non-Goals
  • 50 IETF RFCs — Technical standards with Motivation, Proposal, Constraints, Reasoning
  • Only documents with all 6 required sections included
  • Templates, READMEs, and administrative files excluded

Q&A Generation

  • 100 total Q&A pairs
  • 2 questions per KEP (alternating sections)
  • 1 question per RFC (rotating across 4 section types)
  • Expected phrases extracted from actual section content
  • Semantic match threshold: ≥ 0.80 cosine similarity via all-mpnet-base-v2

Strategies Evaluated

A — Raw Context (Baseline)

Full document passed as-is. Semantic ceiling. No compression applied. Establishes maximum possible recall.

B — BM25

Top-N sentences by BM25Okapi keyword score. Fast, lexical-only, no semantic or structural awareness.

C — Vector Search

Top-N sentences by cosine similarity to query embedding (all-mpnet-base-v2). Semantic but structurally blind.

D — SCRE

Full pipeline: semantic classification + dual graphs + hybrid scoring + reasoning chain expansion. The subject of this report.

Key Findings

✅
SCRE preserves 84.36 SPS while compressing 80.5% of the token volume.

It closes 58% of the gap between BM25/Vector and Raw Context, while still achieving substantial token reduction. It's the only strategy that makes a meaningful dent in Reasoning Graph Recall (12.45% vs <0.05% for both baselines).

⚠️
Dependency Recall is 0.0% for SCRE.

Cross-document citation links (e.g., RFC referencing another RFC) are not yet modeled in the reasoning graph. This is a known limitation and a planned improvement — cross-document entity resolution is on the roadmap.

💡
Why BM25/Vector score higher SER than SCRE.

SER = SPS / avg_tokens. BM25 returns 92 tokens vs SCRE's 518 — so BM25 gets a misleadingly high SER even though its 92 tokens are informationally sparse. SER rewards token efficiency but doesn't capture semantic density. SCRE's 518 tokens are all semantically selected; BM25's 92 are keyword accidents.

📏Metric Definitions

How each benchmark metric is calculated.

SPS — Semantic Preservation Score

SPS = (reasoning_recall × 0.30 + constraint_recall × 0.25 + decision_traceability × 0.20 + workflow_integrity × 0.15 + general_retention × 0.10) × 100

Weighted composite reflecting SCRE's philosophy: reasoning and constraints are more important than general content retention.

📖API Reference

Core methods of the SCRE class.

SCRE(model, db_path, extractors)

Initialize the engine. db_path=":memory:" for ephemeral use; pass a file path for persistent cross-session storage. extractors is a pluggable list of UnitExtractorStrategy instances.

ingest(text, document_id)

Parse, classify, embed, and store a document in the SQLite memory store. No-op if the document_id is already ingested (idempotent). Must be called before retrieve().

retrieve(query, document_id, max_sentences, context_window, min_tokens, max_tokens)

Score and retrieve the best-matching semantic units for a given query. Returns {"context": str, "metadata": dict}. All scoring, graph traversal, and expansion happens here.

reduce(text, query, max_sentences, context_window, min_tokens)

Legacy convenience wrapper. Computes a deterministic document_id from an MD5 hash of the text, calls ingest() then retrieve(). Equivalent to the two-step API in a single call.
